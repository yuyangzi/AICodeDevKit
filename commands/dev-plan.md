# 指令

将已通过审查的设计文档（Spec）转化为**可执行的实施计划**。全程使用中文思考和交流。

## 前置条件

仅在用户明确要求"编写实施计划"或设计文档已审查通过后执行。如果用户未提供 Spec 路径，先确认："请提供要转化成计划的设计文档路径。"

## 第一步：读取 Spec 并收集代码上下文

1. 读取指定的 Spec 文档（`docs/superpowers/specs/`）
2. 并行执行以下探索：
   - **explore agent × 2** — 搜索本次改动涉及的所有代码文件、现有测试模式、import 依赖
   - 直接读取关键文件以获取精确行号和函数签名

## 第二步：编写实施计划

调用 `writing-plans` skill，按以下结构编写计划文档到 `docs/superpowers/plans/YYYY-MM-DD-{slug}.md`：

### Plan 结构要求

```
1. 目标与范围（1 段）
   - 本计划做什么、不做什么

2. 依赖关系图
   - Mermaid 图展示 Task 执行顺序

3. 任务列表（N 个 Task）
   每个 Task 包含：
   ├── 目标：单一定向，可独立验证
   ├── 涉及文件：精确列表
   ├── 完整代码片段：可直接复制执行（非伪代码）
   ├── 验证命令：pytest/curl/bash 精确命令 + 预期输出
   └── Commit 消息

4. 测试策略
   - 后端：TDD 先行（先写测试 → 实现 → 验证）
   - 前端：组件测试在实现后补
   - 集成测试：部署后手动验证清单

5. 风险与回滚
   - 可预见的失败点和应对策略
   - Git revert 步骤
```

### 质量要求（自审清单）

- [ ] 每个 Task 的验证命令精确可复制，包含预期输出
- [ ] 代码片段中无 `...`、`// TODO`、`# implement later` 等占位符
- [ ] 所有函数签名、行号、import 路径与实际代码一致
- [ ] Spec 的每项需求至少被一个 Task 覆盖
- [ ] Task 粒度适中（5-15 分钟完成），可独立提交
- [ ] 依赖图正确标注并行 vs 串行关系

## 第三步：自审并交付

1. 逐项执行自审清单
2. Github commit 保存计划文档
3. 呈现执行概览（N 个 Task，M 波并行）+ 提示用户：

   > "计划就绪。可以执行 `/plan-review` 审查计划，或直接执行 `/esdp` 开始实施。"
