# 指令

对当前未提交的代码变更进行**轻量级审查**。全程使用中文思考和交流。

## 定位

| 命令 | 深度 | 时机 |
|------|------|------|
| `/review-changes`（本命令） | 轻量（< 2 分钟） | 每个 Task 提交前 |
| 内置 `/review-work` | 重型（5 Agent，10+ 分钟） | 重大功能完成、合并前 |

## 审查流程

### 1. 获取变更范围

```bash
git diff --stat            # 变更文件列表
git diff                   # 完整 diff
git log --oneline -5       # 最近提交
```

### 2. 执行诊断

并行运行：
```bash
# 类型/Lint 检查（按项目选择）
lsp_diagnostics 对每个变更文件
# 测试
pytest tests/ -x --tb=short  # 或 npm test
```

### 3. 逐文件审查清单

对每个变更文件，回答以下 5 个问题：

| # | 检查项 | 信号 |
|---|--------|------|
| 1 | **类型安全** | `as any`、`@ts-ignore`、`@ts-expect-error`？ |
| 2 | **遗留调试代码** | `console.log`、`print(`、`TODO`、注释掉的代码？ |
| 3 | **破坏性变更** | 函数签名改变、API 路径改变、返回值类型改变？如有 → 列出所有调用点 |
| 4 | **错误处理** | 空 `catch(e) {}`、静默吞异常？ |
| 5 | **与设计一致** | 改动是否偏离 Spec/Plan 的意图？ |

### 4. 输出格式

```
## 变更审查

### 范围
- 变更文件: N 个 (+X / -Y 行)
- 变更类型: [新功能 / Bug修复 / 重构]

### 诊断
- LSP: ✅ 0 errors / ❌ N errors
- 测试: ✅ N passed / ❌ N failed

### 逐文件审查
| 文件 | 类型安全 | 调试残留 | 破坏性 | 错误处理 | 一致性 |
|------|---------|---------|--------|---------|--------|
| xxx.py | ✅ | ✅ | N/A | ✅ | ✅ |

### 发现
- 🟢 全部通过 — 可以提交
- 🟡 N 个建议（非阻塞）
- 🔴 N 个问题（提交前修复）
```

### 5. 判定

| 结果 | 行动 |
|------|------|
| 全部 ✅ | "审查通过。可以提交。" |
| 🟡 建议 | 呈现建议 → "要修复这些建议项吗？" |
| 🔴 问题 | **直接修复**（不等用户确认），修复后重新审查 |

## 边界

- **仅审查未提交的变更**。不检查预有错误。
- **5 分钟内完成**。如果变更量过大（>10 文件），提示用户考虑 `/review-work`。
- **不替代 Code Review**。这是自动化自查，人的审查仍然需要。
