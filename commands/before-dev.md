# 指令

首先根据用户输入**分类任务类型**，然后按对应路径执行。全程使用中文思考和交流。

## 第一步：任务分类

根据用户意图将任务归为以下四种之一：

| 类型 | 识别特征 | 示例 |
|------|---------|------|
| **Bug 修复** | "异常"、"报错"、"不工作"、"修一下" | "提交反馈功能异常，帮我检测修复下" |
| **功能开发** | "添加"、"实现"、"做一个"、"新功能" | "实现后台管理中的反馈管理 tab" |
| **架构/重构** | "重构"、"移动"、"拆分"、"统一到" | "将后端代码统一移动到 backend 文件夹下" |
| **调查/讨论** | "分析"、"定位"、"探讨"、"讨论设计" | "分析定位下这个异常" |

## 第二步：按类型走不同路径

### 🐛 Bug 修复路径

1. 调用 `systematic-debugging` skill 系统化排查
2. 定位根因后**直接给出修复方案**（无需写完整 spec）
3. 如果修复涉及 3+ 个文件 → **简短设计文档**（5 分钟内写完）
4. 回归测试验证

### 🚀 功能开发路径

1. 调用 `brainstorming` skill 深度思考需求
2. 调用 `grill-me` skill 确认模糊点（**2-4 个关键问题**，不需要穷尽）
3. 提出 2-3 种方案对比，推荐其一
4. 获确认后编写**设计文档（Spec）**到 `docs/superpowers/specs/`
5. 自审通过后**提示用户**："审核通过后我进入实施计划编写"——用户可调用 `/plan-review` 审查 spec

### 🏗️ 架构/重构路径

1. 调用 `brainstorming` skill 理解动机和约束
2. 并行 explore agent 摸清代码现状（import 依赖、路径引用、Shell 脚本）
3. 提出 2-3 种方案（含风险/收益对比），推荐其一
4. 分节呈现设计（每节确认后再继续），降低认知负担
5. 编写**设计文档（Spec）**到 `docs/superpowers/specs/`
6. 提示用户审查

### 🔍 调查/讨论路径

1. 调用 `brainstorming` skill 明确调查方向
2. 并行 explore agent 搜码（必要时配 librarian 查外部资料）
3. **直接给出分析结论**（无需 spec/plan）
4. 如果调查发现需要修复 → 转为 Bug 修复或功能开发路径

## 术语统一

| 产物 | 位置 | 后续命令 |
|------|------|---------|
| **设计文档（Spec）** | `docs/superpowers/specs/YYYY-MM-DD-{slug}.md` | `/plan-review` 审查 |
| **实施计划（Plan）** | `docs/superpowers/plans/YYYY-MM-DD-{slug}.md` | `/esdp` 执行 |
